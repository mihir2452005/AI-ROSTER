# Deploying RoastGPT to production

This guide walks you through deploying:
- **Backend** (FastAPI + PostgreSQL) to [Render](https://render.com)
- **Frontend** (Next.js) to [Vercel](https://vercel.com)

No Docker is used. Both platforms support native runtimes.

---

## 0. Prerequisites

- A GitHub repository with this code pushed (recommended even if you deploy from CLI)
- A [Render](https://render.com) account
- A [Vercel](https://vercel.com) account
- A [Razorpay](https://razorpay.com) account (test keys are fine for staging)
- A custom domain is optional but recommended for production CORS

---

## 1. Deploy the backend to Render

### 1.1. Create the infrastructure from `render.yaml`

The repo ships with a [render.yaml](../render.yaml) blueprint. Render reads it
when you create a new "Blueprint" instance.

1. Go to **Dashboard → New → Blueprint**.
2. Connect your GitHub repo.
3. Render detects `render.yaml` and shows the planned services:
   - `roastgpt-api` (web service, Python)
   - `roastgpt-db` (PostgreSQL)
4. Click **Apply**. Render creates both, runs the build, and starts the service.

### 1.2. Fill in the secrets

In **Dashboard → roastgpt-api → Environment**, add:

| Key                       | Value                                                  |
| ------------------------- | ------------------------------------------------------ |
| `DATABASE_URL`            | *(copy from `roastgpt-db` → Internal Database URL)*   |
| `JWT_SECRET_KEY`          | *(already auto-generated by `generateValue: true`)*    |
| `ADMIN_API_KEY`           | *(already auto-generated)*                             |
| `RAZORPAY_KEY_ID`         | `rzp_test_xxx` from your Razorpay dashboard             |
| `RAZORPAY_KEY_SECRET`     | the matching secret                                    |
| `RAZORPAY_WEBHOOK_SECRET` | a strong random string you also paste into Razorpay    |
| `ALLOWED_ORIGINS`         | `https://<your-vercel-app>.vercel.app`                 |

Click **Save Changes**. Render redeploys automatically.

### 1.3. Bootstrap your first admin

Once the deploy is green, open the **Shell** tab in Render and run:

```bash
ADMIN_EMAIL=you@example.com ADMIN_PASSWORD=YourStrongPass123 \
    python -m scripts.bootstrap_admin
```

This creates (or upgrades) the user and grants `is_admin=True`. You can now
log in at `/login` and access `/admin`.

### 1.4. Configure the Razorpay webhook

In Razorpay Dashboard → Webhooks → Add new:

- **URL**: `https://<your-render-app>.onrender.com/api/payments/webhook`
- **Active events**: `payment.captured`, `payment.failed`, `subscription.cancelled`
- **Secret**: paste the same value you put in `RAZORPAY_WEBHOOK_SECRET`

Razorpay will start sending events that activate subscriptions on your server.

### 1.5. Smoke test

```bash
curl https://<your-render-app>.onrender.com/api/health
# {"status":"ok","library_loaded":true,"roasts":...,"personalities":...,"intents":...}
```

## 2. Deploy the frontend to Vercel

### 2.1. Connect the repo

1. Go to **Vercel → Add New → Project**.
2. Import the same GitHub repo.
3. Set **Root Directory** to `frontend`.
4. Framework: Next.js (auto-detected).

### 2.2. Environment variables

In **Settings → Environment Variables**, add:

| Key                    | Value                                          |
| ---------------------- | ---------------------------------------------- |
| `NEXT_PUBLIC_API_URL`  | `https://<your-render-app>.onrender.com`       |

Apply to **Production**, **Preview**, and **Development**.

### 2.3. Deploy

Click **Deploy**. Vercel builds the Next.js app and gives you a public URL.

### 2.4. Update backend CORS

Add your new Vercel domain to `ALLOWED_ORIGINS` on Render (comma-separated):

```
https://<your-app>.vercel.app,https://www.your-custom-domain.com
```

Render redeploys automatically.

---

## 3. Custom domains (optional but recommended)

### Backend (Render)

- Settings → Custom Domains → add `api.your-domain.com`.
- Render gives you a CNAME. Add it at your DNS provider.
- HTTPS is automatic.

### Frontend (Vercel)

- Settings → Domains → add `your-domain.com` and `www.your-domain.com`.
- Vercel gives you the A / CNAME records. Add them.
- HTTPS is automatic.

Then update `ALLOWED_ORIGINS` on Render and `NEXT_PUBLIC_API_URL` on Vercel
to use the new domains, redeploy both.

## 4. Production checklist

- [ ] `JWT_SECRET_KEY` is a strong random string (Render auto-generated it)
- [ ] `ADMIN_API_KEY` is a strong random string (Render auto-generated it)
- [ ] `ALLOWED_ORIGINS` does **not** include `http://localhost:3000` in production
- [ ] `RAZORPAY_KEY_ID` starts with `rzp_live_` (not `rzp_test_`) for real payments
- [ ] Razorpay webhook is configured and the secret matches `RAZORPAY_WEBHOOK_SECRET`
- [ ] First admin user has been bootstrapped and you can log in
- [ ] `/api/health` returns `library_loaded: true` on the deployed service
- [ ] `/admin` loads the dashboard with your admin user
- [ ] `/pricing` loads the 3 plans

## 5. Security posture (built-in)

These defenses are on by default. **Don't disable them in production.**

- **Secret validation at startup.** The backend refuses to boot in
  production (`ENVIRONMENT=production`) if `JWT_SECRET_KEY` or `ADMIN_API_KEY`
  are missing or still the dev defaults. Override only with
  `ALLOW_INSECURE_DEFAULTS=1` (test envs only).
- **JWT claims.** Tokens carry `sub` (user id), `exp`, `iat`, `jti`, and `ver`
  (current `token_version`). The backend rejects tokens with a `ver` that
  doesn't match the user's `token_version`. Changing password, logging out,
  admin deactivation, or admin demotion all bump `token_version`, instantly
  invalidating the user's other sessions.
- **Algorithm allow-list.** Only `HS256` / `HS384` / `HS512` are accepted; any
  other value crashes startup. Decode also passes `leeway=30s` to handle
  small clock skew.
- **Login timing.** When the email is unknown, the backend runs a bcrypt
  verify against a dummy hash before returning 401, to make user-enumeration
  via timing impractical.
- **CORS.** `ALLOWED_ORIGINS` is the explicit allow-list (comma-separated).
  Wildcards are never used with credentials.
- **Rate limiting.** Per-IP fixed window. The IP is only read from
  `X-Forwarded-For` if the **direct peer** is in `TRUSTED_PROXIES` (defaults
  to `127.0.0.1/32,::1/128`). In Render, set `TRUSTED_PROXIES` to
  `10.0.0.0/8` (Render's internal CIDR) so `X-Forwarded-For` is honoured.
- **Body size cap.** Requests over `MAX_BODY_BYTES` (default 5 MB) get a 413
  before any route runs.
- **Token storage on the client.** Tokens live in `sessionStorage`, not
  `localStorage`. This means they vanish when the tab closes, which is the
  trade-off for a same-origin SPA. For the strictest setup, move to
  httpOnly + Secure + SameSite cookies.
- **Free-tier atomicity.** The free counter is incremented with a single SQL
  `update(...).where(free_messages_used < 5)`. Two parallel requests can
  never both pass the gate.
- **Webhook idempotency.** The `payments` table has a unique index on
  `razorpay_payment_id`. Razorpay may redeliver events; duplicates are
  rejected, not double-applied.
- **Subscription uniqueness.** A user can only have one `active` or
  `past_due` subscription at a time. The backend enforces this on
  `create-order`, `grant-subscription`, and the webhook.
- **Stored XSS.** All user-supplied text (name, username, opener, closer,
  chat content) is run through `sanitize_text` before persisting or
  returning. HTML angle brackets and control characters are stripped.
- **Public leaderboard display names.** Stripped of HTML, control characters,
  and URLs (replaced with `[link removed]`).
- **Email masking.** Every public-facing email is masked
  (`al***@example.com`).
- **In-memory session caps.** `ROASTGPT_MAX_SESSIONS` and
  `ROASTGPT_MAX_MEM_USERS` (default 10 000 each) prevent memory exhaustion;
  the oldest live session is evicted on overflow.

### Known gaps (TODO)

- **No Alembic migrations yet.** Schema is created via `Base.metadata.create_all`
  on boot. Fine for greenfield; switch to `alembic` before the first
  zero-downtime deploy.
- **No password reset / email verification / account deletion** endpoints.
  Admins can deactivate users, but a user cannot self-delete or reset.
- **No CSRF protection** on cookie-based flows (not relevant yet; we're
  using `sessionStorage` Bearer tokens).
- **No 2FA / TOTP.**

## 6. Local development (for reference)

```bash
# Backend
cd backend
python -m venv .venv && source .venv/bin/activate   # or .venv\Scripts\activate on Windows
pip install -r requirements.txt
cp ../.env.example .env                             # edit values
uvicorn main:app --reload --port 8000

# Frontend (in a second terminal)
cd frontend
npm install
echo "NEXT_PUBLIC_API_URL=http://localhost:8000" > .env.local
npm run dev
```

Open http://localhost:3000. The frontend talks to the backend on 8000.

## 7. Cost notes

- Render free tier services spin down after 15 min of inactivity. The first
  request after a pause can take 30+ seconds. For a real product, use the
  $7/mo starter plan to stay always-on.
- Render Postgres free tier is 1 GB and is wiped after 90 days. Use the
  $7/mo starter for production.
- Vercel free tier is generous for Next.js — you only pay once you exceed
  100 GB bandwidth / month.
- Razorpay charges 2% per successful transaction. No monthly fee.
